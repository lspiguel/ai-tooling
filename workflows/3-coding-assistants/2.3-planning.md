# [2.3] Planning / Design — Plan Mode: a detailed implementation plan, peer reviewed

## What this activity is

- **Design and architect** an implementation plan that fulfills the intent and AC for a work item. Use **Plan Mode** to build this with the assistant.

- **Design Collaboration Sessions** between team menbers can be carried out to ellaborate on the design.

- **Deliverables** are updating user story/stories with complete:
1. Design notes
2. Comments

## Activities

### Flesh out the plan in Plan Mode

**AI:** Copilot agent mode / Claude Code (Plan Mode)
**Context:** Repo root

```
In plan mode: from this specification, produce a detailed implementation
plan grounded in this repo. For each step: the real solution components
and files it touches (cite paths), the pattern to follow (name an existing
class/component to mirror), and the order of work with its rationale.
Output the plan as markdown for review. Do NOT implement anything yet.
[paste or reference spec]
```

### Peer review the plan

```
Review this implementation plan as a second reviewer — a human has already
reviewed it. Check it against the spec's acceptance criteria and against
the repo: components the plan touches but doesn't mention, missing test or
ALM steps, and sequencing that conflicts with the solution structure.
Report findings only; do not rewrite the plan.
[paste plan]
```

### Sequencing from solution structure

```
Analyze the unpacked solution tree and tell me the dependency order for
delivering these features: which stories touch shared tables or plugins
and therefore conflict if worked in parallel, which are isolated, and a
suggested sprint-by-sprint sequence. Cite the component files that create
each dependency.
[list features/stories]
```

### Risk register grounded in the repo

```
Draft a risk register for this delivery plan. Beyond the standard project
risks, derive technical risks from the repo itself: plugins with no test
coverage in areas we're changing, components with heavy cross-solution
dependencies, legacy patterns (Xrm.Page, sync plugins) adjacent to planned
work. For each: likelihood, impact, early-warning signal, mitigation.
```

### Draft an ADR

```
Draft an ADR in docs/adr/ for the decision to [decision, e.g. "implement
the pricing recalculation as a synchronous plugin rather than a flow"].
Use the standard format: context, decision, alternatives considered
(with the real constraints from this codebase), consequences.
Status: proposed.
```

---

## Review checkpoints

| Output | What the human checks |
|---|---|
| Implementation plan | Reviewed in full **before** it advances to tasking; cited paths and patterns are current |
| Sequencing | Dependencies sanity-checked — repo structure shows coupling, not team constraints |
| ADRs | A human owns the decision; the ADR records it, it doesn't make it |
