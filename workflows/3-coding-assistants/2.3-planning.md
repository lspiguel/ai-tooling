# [2.3] Planning — a real ADO work-item tree, sequenced from solution structure

> **Matrix cell:** [[2.3]](../ai-augmented-d365ce-activity-matrix.md#23-planning) · Column **[3] Coding Assistants & Agentic Environments** · Builds on [B.3](./b.3-initial-setup.md) + [C.3](./c.3-project-setup.md) · Column-1 counterpart: [2.1](../1-general-purpose-assistants/2.1-planning.md)

## What this activity is

Same planning thinking as [2.1] — decompose, sequence, risk, estimation rationale — but the plan **lands as a real Azure DevOps work-item hierarchy via the ADO MCP** instead of a document someone retypes. Two further column-3 advantages:

- **Dependency and sequencing analysis from solution structure.** The unpacked solution tree shows what actually depends on what (which plugins touch which tables, which forms host which web resources), so sequencing is derived, not guessed. **Plugin B — Import Dependency Pre-Flight** (build backlog) feeds this row for cross-solution import ordering.
- **Draft ADRs** for the architecture decisions the plan implies, kept in `docs/` as docs-as-code.

HITL: the assistant proposes the tree and writes it **only after the human approves the draft** — creating dozens of work items is an outward-facing act.

## Activities

### Decompose and propose the work-item tree (dry run)

**AI:** Copilot agent mode / Claude Code
**Context:** Repo root + ADO MCP

```
From this specification, propose an Azure DevOps work-item hierarchy:
epic → features → user stories, with acceptance criteria in Gherkin on
each story. Ground the technical notes in this repo — reference the real
solution components each story touches.
Output the tree as markdown for review. Do NOT create anything in ADO yet.
[paste or reference spec]
```

### Create the approved tree in ADO

```
The tree is approved as amended. Create these work items in Azure DevOps
project [project], area path [area], iteration [iteration], with the
parent-child links as drafted. Report back the created IDs as a table
mapping title → ID → URL.
```

### Sequencing from solution structure

```
Analyze the unpacked solution tree and tell me the dependency order for
delivering these features: which stories touch shared tables or plugins
and therefore conflict if worked in parallel, which are isolated, and a
suggested sprint-by-sprint sequence. Cite the component files that create
each dependency.
[list features/stories]
```

### Risk register grounded in the repo

```
Draft a risk register for this delivery plan. Beyond the standard project
risks, derive technical risks from the repo itself: plugins with no test
coverage in areas we're changing, components with heavy cross-solution
dependencies, legacy patterns (Xrm.Page, sync plugins) adjacent to planned
work. For each: likelihood, impact, early-warning signal, mitigation.
```

### Draft an ADR

```
Draft an ADR in docs/adr/ for the decision to [decision, e.g. "implement
the pricing recalculation as a synchronous plugin rather than a flow"].
Use the standard format: context, decision, alternatives considered
(with the real constraints from this codebase), consequences.
Status: proposed.
```

| Output | What the human checks |

| ADRs | A human owns the decision; the ADR records it, it doesn't make it |
