# [4.3] Implementation / Build — Agent Mode: the core build with a human in the loop

> **Matrix cell:** [[4.3]](../ai-augmented-d365ce-activity-matrix.md#43-implementation--build) · Column **[3] Coding Assistants & Agentic Environments** · Builds on [B.3](./b.3-initial-setup.md) + [C.3](./c.3-project-setup.md) · Column-1 counterpart: [4.1](../1-general-purpose-assistants/4.1-implementation.md)

## What this activity is

**Agent Mode — automated implementation with a human in the loop.** This is the heavy lifting, where the coding assistant earns column 3's existence: plugins, PCF controls, web resources, Azure Functions, and pipeline YAML, built inside the grounded repo with build/test feedback loops. The assistant also covers the snippet work from [4.1] (Power Fx, FetchXML, OData), but here it can verify against the unpacked solution and live schema instead of trusting your paste. Where skills and tooling are unavailable — work that must be done by hand in the maker portal — fall back to step-by-step configuration instructions.

**The 2026 Copilot split matters — choose the mode per task:**

- **Agent mode** — synchronous, in-IDE, multi-file edits **with you watching**. Use for plugins, PCF, refactors — anything where you steer.
- **Coding agent** — asynchronous, cloud, **returns a PR**. Use for well-specified, self-contained tasks you can review later (which is why [3.3](./3.3-tasking.md) pre-flights task descriptions); budget premium requests — a complex task can burn 10–30.

Environment gestures (import, publish, auth) go through **pac CLI** or the **Dataverse coding-agent plugin** (preview) rather than the browser.

This file holds the implementation-phase prompts organized by component type; the [column-3 README](./README.md) keeps additional prompts not yet mapped to a phase. The [prompt patterns appendix](./README.md#appendix-prompt-patterns-that-work-well) (anchor to existing code, constrain scope, separate concerns) applies to everything below.

## Activities

### Plugins and Custom APIs

**AI:** GitHub Copilot (Visual Studio) · **Context:** plugin solution with existing samples · More (mock setup, error codes): [README § Plugin and Custom API Development](./README.md#plugin-and-custom-api-development)

**Generate from an existing pattern** (the anchor-to-existing-code move):
```
Add a plugin similar to AccountUpdatePostOpPlugin but that triggers on
[TableName] Delete, Pre-Validation stage. It should query for active
related [RelatedTable] records and throw an InvalidPluginExecutionException
if any exist, preventing the deletion.
```

**Enforce naming and layering:**
```
Rename the class to follow the convention [EntitySchemaName][Message][Stage]Plugin
and add the registration comment block to the class summary.
```
```
Move the RetrieveMultiple calls into a static class in the Queries namespace
and folder. Each method should return an EntityCollection and accept an
IOrganizationService parameter.
```

**Unit tests, in the house style:**
```
Create a unit test class for plugin [PluginClassName]
inheriting from PluginTestHelperBase and following the style in [ExistingTestClass].
Include tests for: successful execution, each validation failure path,
and any branching logic.
```

**Debugging:**
```
Here is a plugin trace log. Identify the root cause of the failure,
the method and line where it occurred, and suggest a fix.
[paste trace log]
```

### Web resources (JavaScript)

**Context:** open code file / Web Resources folder · More (code explanation, ribbon commands): [README § JavaScript and Web Resources](./README.md#javascript-and-web-resources)

**Generate a form event handler:**
```
Write a Dynamics 365 CE JavaScript form OnLoad function for the [TableName] table
that hides the field [fieldname] when [otherfield] equals [value].
Use form or global context (not Xrm.Page) and follow strict mode and namespace conventions.
```

**Modernize legacy patterns:**
```
This code uses XMLHttpRequest and var declarations.
Rewrite it using fetch, const/let, and async/await,
without changing the business logic.
```

### PCF controls

**Context:** PCF index file · More (hooks migration, dead code, mock harness): [README § PCF Controls](./README.md#pcf-controls)

**Generate a control from a description:**
```
Create a PCF field control for Dynamics 365 CE that renders a read-only
color-coded badge based on the value of a OptionSet field.
Use React and TypeScript. Follow the standard PCF manifest and index structure.
```

**Decompose a monolith:**
```
Divide this code into components following React best practices.
Identify logical UI sections and extract each into its own functional component
with typed props.
```

### Azure Functions

**Context:** Function project in Visual Studio · More (config class, logging, JSON-output testing, dynamic fields): [README § Azure Functions](./README.md#azure-functions)

**Scaffold with Dataverse connectivity:**
```
Generate an Echo function and a WhoAmI function for a .NET 8 Isolated
Azure Function project that connects to Dataverse.
WhoAmI should use ServiceClient with credentials from environment variables:
InstanceUri, ClientId, ClientSecret, TenantId.
```

**Resilience:**
```
Wrap the Dataverse ServiceClient calls in this function with a retry policy
using Polly: 3 retries with exponential backoff starting at 2 seconds,
retrying on HttpRequestException and ServiceException.
```

### Pipelines

```
Generate an Azure DevOps YAML pipeline to build and deploy
a .NET 8 Isolated Azure Function to a specific deployment slot
using a service connection. The pipeline should be manual-trigger only,
restore NuGet packages, build in Release mode, and deploy using zipDeploy.
```
