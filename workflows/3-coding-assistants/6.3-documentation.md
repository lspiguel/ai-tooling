# [6.3] Documentation / Information Sharing — automated docs: technical docs from code, wiki updates

> **Matrix cell:** [[6.3]](../ai-augmented-d365ce-activity-matrix.md#63-documentation--information-sharing) · Column **[3] Coding Assistants & Agentic Environments** · Builds on [B.3](./b.3-initial-setup.md) + [C.3](./c.3-project-setup.md) · Column-1 counterpart: [6.1](../1-general-purpose-assistants/6.1-documentation.md)

## What this activity is

- **Update people-facing documentation**: user guides, runbooks, release notes, and training material.

- **Update deployment checklists** to capture required tasks that will be needed on upper environment when an increment is delivered.

- **Update live documentation** in the form of wikis.

**Automated docs.** AI writes **functional** and **technical** documentation, generated from what's actually in the repo rather than from memory: component docs, ADRs, solution summaries, and automated wiki updates, published alongside user-facing material — plus **runbook review**, checking the hand-written runbooks against what the repo and pipelines actually do.

**Keep the repo grounding current.** If no automated ALM is in plase, re-export + unpack the solutions so the commit diff documents what changed.

## Activities

### Document a table's customizations

**AI:** GitHub Copilot Chat
**Context:** solution folder open in VS Code

```
Document customizations done on the [TableName] table, explaining purpose,
relationships to other tables, and any notable business logic in forms or views.
```

### Generate a full solution summary

```
Summarize all tables in this solution, their purpose, and which other tables
they relate to. Format the output as a markdown table.
```

### List flows by trigger (integration/runbook input)

```
Get me a list of all flows that are triggered from [TableName].
I need the list only, not code.
```

### Explain a security role in plain English

```
Explain what this security role allows a user to do in plain English,
grouping privileges by functional area.
```

### Document code in place (JSDoc)

```
Add a JSDoc comment header to each function explaining what it does,
its parameters, and return value. Do not change any existing code.
```

### Generate a technical doc from a component

```
Write a technical doc page (docs/[component].md) for [component/plugin/
PCF control]: what it does, the tables and messages it's registered on
(from the actual step definitions in this repo), its dependencies,
configuration, and known limitations. Cite file paths so the doc is
verifiable. Format for an Azure DevOps wiki.
```

### Review a runbook against the repo

```
Review docs/runbooks/[runbook].md against this repository and its pipelines.
Flag steps that no longer match reality: renamed solutions or components,
changed build/test/deploy commands, and missing pre- or post-deployment
checks that the pipeline now performs. Report findings only; do not rewrite.
```

### Finalize ADRs

ADRs drafted in [2.3](./2.3-planning.md) get their status updated (proposed → accepted/superseded) as decisions land; the assistant keeps the index in `docs/adr/` consistent.

### Refresh the grounding (same pass, every time)

1. **Re-export + unpack** the solutions ([C.3 §5](./c.3-project-setup.md#5-the-solution-exportunpack-script)); the commit diff documents what changed.
2. **Pick up the regenerated context packs** from the column-1 re-export ([6.1](../1-general-purpose-assistants/6.1-documentation.md) / [C.1 §9](../1-general-purpose-assistants/c.1-project-setup.md#9-refresh-cadence)) and commit them to `/context/`.
