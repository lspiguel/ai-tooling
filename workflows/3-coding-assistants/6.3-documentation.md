# [6.3] Documentation / Information Sharing — technical docs from code, living instruction files

> **Matrix cell:** [[6.3]](../ai-augmented-d365ce-activity-matrix.md#63-documentation--information-sharing) · Column **[3] Coding Assistants & Agentic Environments** · Builds on [B.3](./b.3-initial-setup.md) + [C.3](./c.3-project-setup.md) · Column-1 counterpart: [6.1](../1-general-purpose-assistants/6.1-documentation.md)

## What this activity is

Column 3 owns the **technical** documentation, generated from what's actually in the repo rather than from memory: component docs, ADRs, and solution summaries, published to ADO wikis alongside the column-1 user-facing material. Two D365-specific moves close the loop:

1. **Re-run the Context Exporter** so the schema packs are regenerated as living documentation — the same artifact that grounds column 1 *is* the data-model doc. The **Context Exporter CI refresh** (build backlog) is the headless version: a pipeline run that commits regenerated packs.
2. **Keep the instruction files honest.** `copilot-instructions.md` / `CLAUDE.md` / `AGENTS.md` rot like any doc; refresh them from the actual repo (`/init` or equivalent) in the same pass, or the assistant's grounding quietly degrades.

## Activities

### Document a table's customizations

**AI:** GitHub Copilot Chat
**Context:** solution folder open in VS Code

```
Document customizations done on the [TableName] table, explaining purpose,
relationships to other tables, and any notable business logic in forms or views.
```

### Generate a full solution summary

```
Summarize all tables in this solution, their purpose, and which other tables
they relate to. Format the output as a markdown table.
```

### List flows by trigger (integration/runbook input)

```
Get me a list of all flows that are triggered from [TableName].
I need the list only, not code.
```

### Explain a security role in plain English

```
Explain what this security role allows a user to do in plain English,
grouping privileges by functional area.
```

### Document code in place (JSDoc)

```
Add a JSDoc comment header to each function explaining what it does,
its parameters, and return value. Do not change any existing code.
```

### Generate a technical doc from a component

```
Write a technical doc page (docs/[component].md) for [component/plugin/
PCF control]: what it does, the tables and messages it's registered on
(from the actual step definitions in this repo), its dependencies,
configuration, and known limitations. Cite file paths so the doc is
verifiable. Format for an Azure DevOps wiki.
```

### Finalize ADRs

ADRs drafted in [2.3](./2.3-planning.md) get their status updated (proposed → accepted/superseded) as decisions land; the assistant keeps the index in `docs/adr/` consistent.

### Refresh the grounding (same pass, every time)

1. **Re-export + unpack** the solutions ([C.3 §6](./c.3-project-setup.md#6-the-solution-exportunpack-script)); the commit diff documents what changed.
2. **Re-run the Context Exporter** and commit the packs to `/context/` (and re-upload to the column-1 Project per [C.1 §9](../1-general-purpose-assistants/c.1-project-setup.md#9-refresh-cadence)).
3. **Refresh the instruction files:**

```
Re-read this repository and update copilot-instructions.md / CLAUDE.md and
AGENTS.md to match reality: new projects, renamed folders, changed build or
test commands, new conventions that appeared in the code. Show me the diff —
keep each file under ~1,000 words and do not drop the standing rules
(prefix, no early binding, data boundary, describe-via-MCP).
```

| Output | What the human checks |

| Generated docs | Spot-check cited paths and registrations against the repo |
| Instruction-file diffs | Standing rules survived; nothing client-specific leaked into the template repo copy |
| Regenerated packs | LEGAL.md notice present; correct environment; export date updated |
