# [5.3] Validation / Peer Review — conventions-aware code review, drift detection, PR hygiene

> **Matrix cell:** [[5.3]](../ai-augmented-d365ce-activity-matrix.md#53-validation--peer-review) · Column **[3] Coding Assistants & Agentic Environments** · Builds on [B.3](./b.3-initial-setup.md) + [C.3](./c.3-project-setup.md) · Column-1 counterpart: [5.1](../1-general-purpose-assistants/5.1-validation-peer-review.md)

## What this activity is

- **Peer review** on pull requests, review of design docs (at the description/narrative level), check unit-test coverage, sanity-checking an approach, and cross-checking a delivered implementation *narrative* against the requirements doc.

- **Executing test plans** manually from acceptance criteria, may include integration testing, and automated testing -only if it uses distinct tooling from building-.

- **HITL discipline:** an AI review comes *after* the human has already reviewed, as a final detail-catcher — never as the first pass. First-pass AI feedback anchors the human reviewer and defeats the purpose of peer review.

- **Ask Mode with an adversary reviewer prompt** for automated review: a consistent, explicit rule set (StyleCop, prefixes, no early binding, fixture coverage) gives consistent, conventions-aware review — better than ad-hoc prompting the same rules each time.

## Activities

### Code review (after the human pass)

**AI:** GitHub Copilot Chat / reviewer persona
**Context:** the PR diff / plugin class

**Best-practices review:**
```
Review this plugin class for the following: unnecessary catch blocks that
rethrow without adding information, missing null checks,
any hardcoded GUIDs or magic strings that should be constants,
and any N+1 query patterns.
```

**Style conformance:**
```
Correct this code to conform to StyleCop style,
add missing documentation comments.
```

**JS quality:**
```
What JS styling and quality rules from SonarQube or ESLint
could be applied to this code? List each issue with the line reference.
```

**Dead code sweep:**
```
Is there any dead code, commented-out code, or duplicate functions
across these files? List findings with file and line number.
```

### Unit-test coverage check

```
Compare the unit tests on this branch against the unit test conditions
defined in task [ID]'s description. List: conditions with no covering test,
tests that assert nothing from the conditions, and branches in the changed
code with no test at all.
```

### Configuration review

**Naming conventions across the solution:**
```
Review the customizations in this solution folder.
Flag any table, field, form, view, or flow names that do not follow
the prefix convention "[prefix]_" or that use default system-generated names.
```

**Flows — error handling and performance:**
```
Identify cloud flows in this folder that do not have a "Configure run after"
or scope-based error handling. List them with their trigger.
```
```
Which flows in this folder iterate over collections using Apply to Each?
List them and note if parallel branching is enabled.
```

### Security review

**Context:** security solution folder · More (PowerShell audit script): [README § Security Analysis](./README.md#security-analysis)

```
Analyze the security role definitions in this folder.
Flag any roles that have organization-level access on sensitive tables
such as SystemUser, Team, or BusinessUnit.
```

### Power Pages review

```
Review these Liquid template files and flag any places where user input
is rendered without escaping, or where table permissions are bypassed.
```

### Solution health / technical debt

```
Review this solution repository. Flag: deprecated APIs,
hardcoded environment URLs, plugins running synchronously that could be async,
flows using legacy connectors, and any components using unsupported patterns
for the current Power Platform version.
```

### Drift detection

Run **Plugin A — Component Snapshot & Drift Tracker** to diff the environment against the last snapshot; alternatively, the manual fallback is re-running the export script ([C.3 §5](./c.3-project-setup.md#5-the-solution-exportunpack-script)) and reviewing the git diff — any change nobody committed intentionally *is* the drift report.

### PR description + self-review

```
Generate the PR description for this branch: summary of the change,
components touched (real paths), how it satisfies the story's ACs
(reference [story ID]), test evidence, and anything the reviewer should
look at first. Then do a self-review pass and list the concerns you'd
raise if this were someone else's PR.
```

### Automated PR review via GitHub Actions

Concept and prompt: [README § AI-Powered Code Review (GitHub Actions)](./README.md#cicd-and-automation). Position it as a **post-human** gate per the HITL rule.

## Review checkpoints

| Output | What the human checks |
|---|---|
| Persona/AI findings | Triage each; human decides severity — the AI never blocks or approves a PR |
| Security findings | Verify against the actual org before acting |
| Drift report | Classify each drift: hotfix to backport, or unauthorized change to revert |
