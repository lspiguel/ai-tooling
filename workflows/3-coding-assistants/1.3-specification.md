# [1.3] Specification / Intent — grounded gap analysis and technical specs that name real components

> **Matrix cell:** [[1.3]](../ai-augmented-d365ce-activity-matrix.md#13-specification--intent) · Column **[3] Coding Assistants & Agentic Environments** · Builds on [B.3](./b.3-initial-setup.md) + [C.3](./c.3-project-setup.md) · Column-1 counterpart: [1.1](../1-general-purpose-assistants/1.1-specification.md)

## What this activity is

- **Capturing intent** by describing what will be costructed is expected to provide to the overall system and where necessary describing how that fits on the architecture and design.

- **Turn raw stakeholder input** (meeting notes, emails, workshop whiteboards) into the **epic → feature → user story hierarchy** with acceptance criteria that follow the house templates seeded in the [B.3] Project.

- **Deliverables** are the user story/stories with complete:
1. Title
2. Intent/Description
3. Acceptance Criteria
4. Attachments or reference materials
This would need to be pushed into the Work Management Platform (ADO, Jira, etc) for collaboration and managent.

Write user stories & ACs — concentrating on providing **intent and acceptance criteria** rather than prescribing the implementation. Use **Ask Mode** to explore options for ACs before committing to one redaction. The reconciliation against what exists is **live**, because the assistant is sitting in the grounded repo:

- **`describe` / `search` over Dataverse MCP** gives current schema, not a point-in-time pack.
- The **unpacked solution tree** (`solutions/<Name>/src/`) lets the gap analysis cite real files: the actual plugin steps, form XML, and web resources a change will touch.
- **ADO MCP/CLI** pulls the related existing work items, so the spec doesn't duplicate what's already in the backlog.

## Activities

### Grounded gap analysis for a new requirement

**AI:** Copilot agent mode / Claude Code
**Context:** Repo root, Dataverse MCP + ADO MCP available

```
New requirement: [describe requirement].
Produce a gap analysis:
1. Use Dataverse MCP describe/search to check which tables and columns
   already cover part of this — cite logical names.
2. Search the unpacked solution tree for existing plugins, flows, forms,
   and web resources that implement related behavior — cite file paths.
3. Query ADO for open or closed work items touching the same area.
4. Conclude: what exists, what must change, what is net-new.
Do not propose an implementation yet.
```

### Explore options in Ask Mode

```
In ask mode (no edits): here is the intent and the draft acceptance criteria
for [requirement]. Lay out the implementation options this environment
supports, each option's trade-offs given the unpacked solution and live
schema, and the questions we'd need answered before choosing.
Do not write the spec or any code yet.
```

### Find all references to a field

**AI:** GitHub Copilot Chat
**Context:** Full solution repo

```
Find every place the field [table_fieldname] is used across this solution:
forms, views, flows, JavaScript, plugins, and FetchXML queries.
```

### Assess impact before deleting a table or field

```
I'm planning to remove the field [table_fieldname].
What components in this repository reference or depend on it?
```

### Draft the technical spec

```
Using the gap analysis above, draft a technical specification for
[requirement]. Requirements: name the real entities/attributes (verified
via Dataverse MCP, not assumed), list the exact plugin classes and steps
that change with file paths, list new components with names following the
[prefix]_ naming convention, and state migration and
security-role impacts. Flag any point where the environment and the
unpacked solution disagree (drift) instead of guessing.
```

### Check the backlog for duplication

**Context:** ADO MCP

```
Search our Azure DevOps project for existing work items related to
[feature/table/process]. List ID, title, state, and how each relates to
this new requirement — duplicate, prerequisite, or superseded.
```

## Review checkpoints

| Output | What the human checks |
|---|---|
| Gap analysis | Spot-check the cited schema in the real environment; MCP results current but repo may lag |
| Technical spec | Architecture fit; naming conventions; nothing invented where MCP wasn't consulted |
| Drift flags | Any environment-vs-repo mismatch triggers a re-export ([C.3 §5](./c.3-project-setup.md#5-the-solution-exportunpack-script)) before the spec is finalized |
