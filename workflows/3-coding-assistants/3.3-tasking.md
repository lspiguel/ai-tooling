# [3.3] Tasking — expand the plan: checklists, unit test conditions, ADO tasks via MCP

> **Matrix cell:** [[3.3]](../ai-augmented-d365ce-activity-matrix.md#33-tasking) · Column **[3] Coding Assistants & Agentic Environments** · Builds on [B.3](./b.3-initial-setup.md) + [C.3](./c.3-project-setup.md) · Column-1 counterpart: [3.1](../1-general-purpose-assistants/3.1-tasking.md)

## What this activity is

**Expand the plan** approved in [2.3](./2.3-planning.md) into board-ready work, all grounded:

1. **Task out & prepare checklists** — break the plan into tasks with crisp descriptions and an explicit Definition of Done.
2. **Define unit test conditions** — each task carries the conditions its tests must cover, so testing is decided before the build starts.
3. **Create the work items in ADO via MCP** under the approved stories.
4. **Scaffold branches** following the engagement's branch-naming convention.
5. **Emit task-level technical notes grounded in the repo** — the note on a task references the real files (`solutions/<Name>/src/...`, `src/Plugins/...`) the developer will touch, so nobody starts a task by re-discovering where things live.

Well-grounded task notes are also what make a task delegable to the **asynchronous coding agent** later ([4.3](./4.3-implementation.md)): a task whose note names the exact files and conventions is a task the agent can pick up cold.

## Activities

### Expand the plan into repo-grounded tasks (dry run)

**AI:** Copilot agent mode / Claude Code
**Context:** Repo root + ADO MCP + the approved plan from [2.3](./2.3-planning.md)

```
Expand the approved implementation plan for story [ID/title] into tasks.
For each task include a technical note grounded in this repo:
the files to change (real paths), the pattern to follow (name an existing
class/component to mirror), the unit test conditions the task must cover,
the test approach, and the Definition of Done.
Flag any task where the repo and the plan disagree.
Output as markdown for review — do not create tasks in ADO yet.
```

### Create the approved tasks

```
Approved. Create these as Task work items in ADO, linked as children of
story [ID], iteration [iteration]. Put the technical note (including the
unit test conditions) in the task description. Report the created IDs.
```

### Scaffold the branches

```
Create a branch for story [ID] from main using our convention
[e.g. feature/{storyId}-{kebab-title}], and push it to the remote.
List the branch name(s) created.
```

### Pre-flight a task for the async coding agent

```
Review task [ID]'s description as if you were the coding agent picking it
up with no other context. What's missing or ambiguous? Check: are the file
paths current, is the expected behavior testable, is the DoD explicit,
does it conflict with any open branch? Rewrite the description to close
the gaps.
```

## Review checkpoints

| Output | What the human checks |
|---|---|
| Task breakdown + notes | Cited file paths are current; pattern references appropriate; no invented scope |
| ADO writes / branch pushes | Approved before execution (dry-run first, always) |
