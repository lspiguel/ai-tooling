# [1.1] Specification — stakeholder input to grounded stories

> **Matrix cell:** [[1.1]](../ai-augmented-d365ce-activity-matrix.md#11-specification) · Column **[1] General-Purpose AI Assistants** · Builds on [B.1](./b.1-initial-setup.md) + [C.1](./c.1-project-setup.md) · Column-3 counterpart: [1.3](../3-coding-assistants/1.3-specification.md)

## What this activity is

Specification in column 1 has **two jobs**:

1. **Turn raw stakeholder input** (meeting notes, emails, workshop whiteboards) into structured requirements, user stories, and acceptance criteria that follow the house templates seeded in the [B.1] Project.
2. **Reconcile the ask against what already exists.** This is where the Context Exporter pack earns its keep — the assistant can only flag *"you're asking for a field the OOTB contact already has"* if it can see the data model. Without grounding, the assistant will happily spec a duplicate.

The **AI Assistant** (M365 Copilot–style, scoped to the tenant) adds a third capability: mining Teams, email, and SharePoint for the source material itself — surfacing the requirements conversation you weren't in.

Constraints that apply to every prompt below:

- **Data boundary.** Stakeholder notes and schema extracts go only into the enterprise-tier assistant covered by the engagement's data agreement ([Considerations](../ai-augmented-d365ce-activity-matrix.md#considerations)). Live-data questions route through the tenant-scoped AI Assistant instead.
- **AI drafts; a person decides.** Every story and AC produced here is a draft for the BA/functional consultant to own.

## Activities

### Draft a user story from a requirement

**AI:** Claude / Copilot Chat (grounded Project)
**Context:** Requirement description or meeting notes

```
Write a user story for the following requirement: [describe requirement].
Format it with: Title, Description (As a / I want / So that),
Acceptance Criteria in Gherkin format (Given/When/Then),
and Implementation Notes relevant to Dynamics 365 CE.
```

### Evaluate story quality (INVEST)

```
Evaluate this user story against the INVEST criteria.
Report only criteria that are NOT met, with a short reason for each
and a final summary.
[paste user story]
```

### Reconcile a requirement against the existing model

**Context:** Grounded Project with `EntityDictionary.context.md` / `SolutionInventory.context.md` uploaded

```
Here is a new requirement: [describe requirement].
Using the entity dictionary and solution inventory in this Project's knowledge,
tell me: (1) which existing tables, fields, or processes already cover part of
this ask, (2) what is genuinely new, and (3) any naming or design conflicts
with the existing model. Cite the specific entities and attributes you used.
```

### Impact assessment before a change

```
I need to change the behavior of [process/field/table] in Dynamics 365 CE.
Based on the following solution components, identify which user stories,
flows, plugins, and forms may be affected and need retesting.
[describe the change]
```

> With a grounded Project, replace "[describe the change]" context-pasting with a pointer to the uploaded `SolutionInventory.context.md` pack.

### Mine source material with the AI Assistant

**AI:** Tenant-scoped AI Assistant (M365 Copilot)
**Context:** The tenant's Teams / Outlook / SharePoint

```
Find the discussions from the last [timeframe] about [feature/process name]
across Teams and email. Summarize the requirements that were raised,
who raised them, and any open questions or contradictions between stakeholders.
```

## Review checkpoints

| Output | What the human checks |
|---|---|
| User stories / ACs | Business intent preserved; ACs testable; template followed |
| "Already exists" reconciliation | Verify the cited entities/attributes in the real environment — the pack is a point-in-time snapshot |
| Mined requirements | Attribution correct; nothing taken out of context |
