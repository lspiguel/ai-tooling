# [6.1] Documentation / Information Sharing — user docs, runbooks, and living schema packs

> **Matrix cell:** [[6.1]](../ai-augmented-d365ce-activity-matrix.md#61-documentation--information-sharing) · Column **[1] General-Purpose AI Assistants** · Builds on [B.1](./b.1-initial-setup.md) + [C.1](./c.1-project-setup.md) · Column-3 counterpart: [6.3](../3-coding-assistants/6.3-documentation.md)

## What this activity is

- **Update people-facing documentation**: user guides, runbooks, release notes, and training material.

- **Update deployment checklists** to capture required tasks that will be needed on upper environment when an increment is delivered.

**User-driven docs** are written with AI assistance and published. The assistant drafts from the artifacts already in the grounded Project (specs, design docs, `.context.md` packs)

The D365-specific move in this row: **re-running the Context Exporter is itself a documentation act.** The regenerated `.context.md` packs *are* the living data-model documentation — the same artifact that grounds the assistant. Re-export after material model changes.

## Activities

### Draft a deployment runbook

**AI:** Claude / Copilot Chat

```
Based on the following solution components, write a deployment runbook
for deploying to UAT and then Production.
Include pre-deployment checks, the deployment steps,
post-deployment validation steps, and a rollback procedure.
[list solution components]
```

### Generate a solution changelog

```
Compare the solution manifest files from these two solution exports
and list: components added, removed, or modified between versions.
Format as a changelog.
```

### Write release notes

```
Turn this changelog / list of completed user stories into release notes
for two audiences in one document:
(1) business users — what changed in their daily work, plain language;
(2) admins/support — technical changes, new settings, known issues.
[paste changelog or story list]
```

### Draft a user guide for a feature

```
Write an end-user guide for [feature] using the actual form, view, and
field names from this Project's knowledge. Structure: what it's for,
step-by-step instructions per task, and an FAQ from the edge cases in
the acceptance criteria. Mark places where a screenshot should be inserted.
[paste feature description]
```

### Draft training material

```
Create a training outline for [role, e.g. sales users] covering [features].
For each module: learning objective, a 5–10 minute walkthrough script for
the trainer, and 2–3 hands-on exercises using realistic sample data
(no real customer data).
```

### Refresh the schema packs (living documentation)

Not a prompt — a procedure: re-run the Context Exporter specs per [C.1 §6](./c.1-project-setup.md#6-run-the-specs), review, re-upload to the Project, and update the export date in the Project instructions. Do this as part of closing any story that changed the data model.

## Review checkpoints

| Output | What the human checks |
|---|---|
| Runbook | Steps validated against the actual environment topology before first use |
| User docs / training | Walked through in the real app; names and navigation correct |
| Release notes | No internal detail leaking to business audience |
