# [2.1] Planning — decomposition, sequencing, and the planning artifact

> **Matrix cell:** [[2.1]](../ai-augmented-d365ce-activity-matrix.md#21-planning) · Column **[1] General-Purpose AI Assistants** · Builds on [B.1](./b.1-initial-setup.md) + [C.1](./c.1-project-setup.md) · Column-3 counterpart: [2.3](../3-coding-assistants/2.3-planning.md)

## What this activity is

In column 1 the assistant is a **structured-thinking partner** for planning: decomposing epics into features and stories, sequencing the work, building the risk register, and articulating estimation rationale. The output is a **planning document or deck that a human owns** — the assistant drafts, the delivery lead decides.

The key limitation vs column 3: a general-purpose assistant has **no native work-item integration**. Getting the plan into Azure DevOps happens by **scripting** (a generated import script or CSV) or by hand — contrast with [2.3](../3-coding-assistants/2.3-planning.md), where the ADO MCP writes the work-item tree directly.

Grounding still matters here: sequencing and risk conversations are much better when the assistant can see the `SolutionInventory` pack (what is customized, what integrations exist) rather than reasoning about a generic D365 project.

## Activities

### Break an epic into features and stories

**AI:** Claude / Copilot Chat (grounded Project)
**Context:** Epic description

```
Break this epic into features and user stories for a Dynamics 365 CE implementation.
Maintain the hierarchy: epic → features → user stories.
For each user story include acceptance criteria in Gherkin format.
[paste epic description]
```

### Sequence the work and surface dependencies

```
Here is the feature/story breakdown for [engagement/phase].
Propose a delivery sequence. For each item state: what it depends on
(data model first? security model? integration availability?), what can run
in parallel, and which items carry the most schedule risk. Use the solution
inventory in this Project's knowledge to flag dependencies on existing
customizations.
[paste breakdown]
```

### Draft the risk register

```
Based on the SoW, the architecture document, and this delivery plan,
draft a risk register for this D365 CE engagement.
For each risk: description, likelihood (H/M/L), impact (H/M/L),
early-warning signal, and a concrete mitigation.
Include the standard D365 CE risks (environment strategy, data migration
quality, integration availability, licensing, UAT engagement) only where
the project documents suggest they actually apply.
```

### Estimation rationale

```
For each user story below, draft an estimation rationale: the main cost
drivers (new tables? plugin logic? integration? migration volume?),
a suggested T-shirt size, and the assumptions that would change the estimate.
Do not present these as final estimates — they are input for the team's
estimation session.
[paste stories]
```

### Push the plan into ADO via a generated script

```
Convert this epic → feature → story breakdown into a script that creates
the work items in Azure DevOps with the correct parent links.
Target: [az boards CLI / Azure DevOps REST API / CSV for bulk import].
Project: [project name], Area Path: [area], Iteration: [iteration].
Emit the script only; I will review it before running.
[paste breakdown]
```

---

## Review checkpoints

| Output | What the human checks |
|---|---|
| Decomposition | Feature boundaries match how the team will actually deliver |
| Sequence / dependencies | Validated against real environment + integration availability dates |
| Risk register | No generic filler; every risk traceable to this engagement |
| ADO import script | Dry-run reviewed before execution; work-item types match the process template |

## Open points for iteration

- **Which ADO import mechanism do you actually use** — `az boards` CLI, REST API, or CSV bulk import? The script prompt currently offers all three; I'll pin it once you confirm.
- The sequencing/risk/estimation prompts are new drafts (no README source) — adjust to your house style.
- Does the planning **deck** (as opposed to doc) need its own prompt (e.g. Marp/PowerPoint outline generation)?
