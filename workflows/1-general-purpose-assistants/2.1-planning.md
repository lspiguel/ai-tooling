# [2.1] Planning / Design — design collaboration sessions

> **Matrix cell:** [[2.1]](../ai-augmented-d365ce-activity-matrix.md#21-planning--design) · Column **[1] General-Purpose AI Assistants** · Builds on [B.1](./b.1-initial-setup.md) + [C.1](./c.1-project-setup.md) · Column-3 counterpart: [2.3](../3-coding-assistants/2.3-planning.md)

## What this activity is

**Design collaboration sessions**: architect & design an implementation fulfilling the requirements. In column 1 the assistant is a **structured-thinking partner** — sequencing the work, surfacing dependencies, and building the risk register. The output is a **design artifact that a human owns** — the assistant drafts, the delivery lead decides.

The epic → feature → story breakdown happens upstream in [1.1 Specification / Intent](./1.1-specification.md); estimation and getting tasks onto the board happen downstream in [3.1 Tasking](./3.1-tasking.md). Contrast with [2.3](../3-coding-assistants/2.3-planning.md), where Plan Mode fleshes out the detailed plan grounded in the repo.

Grounding still matters here: sequencing and risk conversations are much better when the assistant can see the `SolutionInventory` pack (what is customized, what integrations exist) rather than reasoning about a generic D365 project.

## Activities

### Design the implementation

**AI:** Claude / Copilot Chat (grounded Project)
**Context:** The stories/ACs from [1.1](./1.1-specification.md), the SoW and architecture doc

```
Here are the user stories and acceptance criteria for [feature/phase].
Propose an implementation design that fulfills them, grounded in the
solution inventory in this Project's knowledge: which existing tables,
processes, and integrations it builds on, which new components it needs,
and the main design decisions with their trade-offs. Present options
where a real choice exists — this is input for a design session,
not a final design.
[paste stories]
```

### Sequence the work and surface dependencies

```
Here is the feature/story breakdown for [engagement/phase].
Propose a delivery sequence. For each item state: what it depends on
(data model first? security model? integration availability?), what can run
in parallel, and which items carry the most schedule risk. Use the solution
inventory in this Project's knowledge to flag dependencies on existing
customizations.
[paste breakdown]
```

### Draft the risk register

```
Based on the SoW, the architecture document, and this delivery plan,
draft a risk register for this D365 CE engagement.
For each risk: description, likelihood (H/M/L), impact (H/M/L),
early-warning signal, and a concrete mitigation.
Include the standard D365 CE risks (environment strategy, data migration
quality, integration availability, licensing, UAT engagement) only where
the project documents suggest they actually apply.
```

## Review checkpoints

| Output | What the human checks |
|---|---|
| Design proposal | Architecture fit; a human owns every design decision — the assistant surfaces options |
| Sequence / dependencies | Validated against real environment + integration availability dates |
| Risk register | No generic filler; every risk traceable to this engagement |

## Open points for iteration

- The design/sequencing/risk prompts are new drafts (no README source) — adjust to your house style.
- Does the design **deck** (as opposed to doc) need its own prompt (e.g. Marp/PowerPoint outline generation)?
