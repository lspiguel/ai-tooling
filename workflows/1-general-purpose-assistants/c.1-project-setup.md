## 1. Prerequisites

| Requirement | Notes |
|---|---|
| XrmToolBox 1.2025.x or later | Installed as part of the machine baseline — `choco install xrmtoolbox -y` (see [B.1](./b.1-initial-setup.md) / [B.3](../3-coding-assistants/b.3-initial-setup.md)). |
| .NET Framework 4.8 | Required by the plugin; present on current Windows. |
| A connection to the **client's** Dataverse environment | System Customizer or System Administrator security role, or at minimum read access to entity metadata, solutions, security roles, and processes. |
| An enterprise/professional-tier AI assistant under the engagement's data agreement | The destination for the packs. Confirm the DPA / zero-retention terms *before* uploading anything. |

Do this once per client environment. If the engagement spans multiple environments (Dev / UAT / Prod), export from the one that is the source of truth for configuration — usually **Dev** — and note the environment URL in the Project so the assistant knows which tenant the grounding describes.

---

## 2. Install the Context Exporter plugin

1. Open **XrmToolBox**.
2. Go to **Tool Library** (Configuration → Tool Library).
3. Search for **D365 CE Context Exporter** (or `D365ContextExporter`).
4. Click **Install**, then restart XrmToolBox when prompted.

---

## 3. Connect to the client environment

1. In XrmToolBox, create or select a **connection** to the client's Dataverse environment.
2. Prefer a **Microsoft 365 / OAuth** connection (interactive sign-in or a service-principal app registration) over stored username/password.
3. Open the **D365 CE Context Exporter** tool against that connection.

> Keep a distinct XrmToolBox connection per client. It prevents accidentally exporting one client's model while pointed at another's tenant.

---

## 4. Deploy the reference configuration to a per-client folder

The exporter works out of a **base directory** that holds the query/template/spec configuration and the rendered output. Use **one base directory per client** so packs never mix.

1. In the tool, click the **folder picker** and choose (or create) an empty folder — e.g. `C:\Clients\<ClientName>\context-exporter\`.
2. On first use the plugin offers to **deploy the reference configuration** into that folder. Accept it. This lays down:

   ```
   context-exporter/
   ├── config/
   │   ├── queries/            FetchXML query files
   │   ├── transformations/    Scriban templates (.sbn)
   │   └── *.context-exporter-config.json   (six sample specs)
   ├── output/                 Rendered .context.md files
   ├── runs/                   Per-run working data
   ├── LEGAL.md                Legal notice prepended to every output
   └── version.txt
   ```

The six specs deployed out of the box:

| Spec | What it captures |
|---|---|
| `EntityDictionary` | All entity definitions with their attributes |
| `SecurityModel` | Security roles and privilege depths |
| `SolutionsReference` | Solution hierarchy and components |
| `FormsAndViews` | Entity forms and views |
| `Optionsets` | Global option sets |
| `SolutionInventory` | Solutions, plugins, flows, environment variables, custom APIs |

The specs are plain JSON and the templates are plain text — edit them if you need to narrow or extend what a pack captures (see the [Context Exporter README](../../tooling/D365ContextExporter/README.md#configuration-reference)).

## 5. Set the legal boundary notice

`config` (base dir) ships a **`LEGAL.md`** that the exporter **prepends to every generated pack**. This is the boundary notice required by the [Considerations](../ai-augmented-d365ce-activity-matrix.md#considerations) — do not skip it.

1. Open `LEGAL.md` in the base directory.
2. Replace the placeholder organisation name with the correct owning party for this engagement, and align the wording with the client's DPA.

   ```markdown
   > [!WARNING]
   > This document was generated by an automated tool and contains confidential
   > business data exported from a Microsoft Dynamics 365 Customer Engagement environment.
   >
   > **Copyright notice:** The content of this file is the exclusive property of
   > <Owning Organisation>. All rights reserved.
   >
   > **Restrictions:** Do not distribute, publish, or share this file outside of
   > <Owning Organisation> without explicit written authorisation. Do not use the
   > contents of this file to train, fine-tune, or evaluate any artificial intelligence
   > or machine learning model.
   >
   > This file is provided solely as grounding context for authorised use within
   > <Owning Organisation>'s internal AI assistant workflows.
   ```

`LEGAL.md` and any custom files are **never overwritten** on a plugin upgrade, so this only needs doing once per client.

## 6. Run the specs

For each spec you want in the grounding set:

1. Select the spec from the **dropdown**.
2. Click **Run**. The plugin executes the spec's queries, renders the Scriban template, and writes `output\<SpecName>.context.md` (with `LEGAL.md` prepended).
3. Repeat for the other specs.

Start with **`EntityDictionary`**, **`SolutionInventory`**, and **`SecurityModel`** — those three give the assistant the most leverage per token (what the tables are, what's customised, who can see what). Add `FormsAndViews`, `Optionsets`, and `SolutionsReference` when the work calls for them.

> **Token budget.** `.context.md` packs can be large. If a pack overruns the assistant's upload/context limit, trim it via the spec's `output.attributeDenyList` or by narrowing the FetchXML rather than truncating the file by hand.

---

## 7. Review the output

Before anything leaves your machine, open each `output\*.context.md` and confirm:

- The **`LEGAL.md` notice is present** at the top of the file.
- The content is **schema/metadata**, not live customer PII. The default specs export configuration, not record data — but if you've customised a spec to pull `webapi` rows, re-check that it doesn't drag in personal data you don't intend to share.
- The environment it describes is the **one you meant** to export from.

---

## 8. Assemble the grounded Project

Create **one grounded Project per client** and seed it:

1. **Create the Project / Custom GPT / Copilot agent** in the enterprise-tier assistant, named for the client.
2. **Upload the `.context.md` packs** from `output\` as project knowledge.
3. **Add the engagement documents** the packs don't cover: the **SoW**, the **solution/architecture** doc, naming and publisher-prefix conventions, and any integration diagrams.
4. **Pin the client's data-handling rules** into the Project's custom instructions — a short paragraph stating the boundary (which tier is approved, what must never be pasted in, that outputs are drafts for human review). This makes the assistant remind *you* when a request would over-share.
5. **Record the source environment URL and the export date** in the instructions so everyone knows how fresh the grounding is and which tenant it reflects.

From here the assistant inherits the client's model on every [1.1] Specification / [2.1] Planning / [5.1] Validation conversation without re-explaining it.

---

## 9. Refresh cadence

The packs are a **point-in-time snapshot**. Re-run the exporter and re-upload when the model has moved materially — after a solution import to Dev, a wave of new tables/fields, or a security-role change. Treat regenerating the packs as the [6.1] Documentation step: the same artifact that grounds the assistant *is* the living data-model doc. Keep the export date in the Project instructions current each time.
