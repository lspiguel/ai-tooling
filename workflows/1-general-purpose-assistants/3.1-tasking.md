# [3.1] Tasking — sprint planning: estimation, assignment, and checklists with a clear Definition of Done

> **Matrix cell:** [[3.1]](../ai-augmented-d365ce-activity-matrix.md#31-tasking) · Column **[1] General-Purpose AI Assistants** · Builds on [B.1](./b.1-initial-setup.md) + [C.1](./c.1-project-setup.md) · Column-3 counterpart: [3.3](../3-coding-assistants/3.3-tasking.md)

## What this activity is

- **Task** out the plan, estimate effort required for each task, discuss resources or team skills needed for each task (i.e. configuration tasks may be done by the functional team, coding tasks may be done by the technical tea but should be reviewed by the functional team).

- **Prioritize** where needed for task ordering and capacity planning.

- **Definition of Done** should be completed at this point -where needed-, generate test plans for overall functionality and define unit tests to be built.

- **Sprint planning**: estimate effort, assign tasks The matrix calls the core gesture **low-tech, high-value**: turn each feature or story into a task checklist with crisp descriptions and an explicit 

The column-1 boundary applies as in Planning / Design: tasks are drafted as text; getting them onto the board is scripting or copy-paste. If the team has coding-assistant tooling, prefer [3.3](../3-coding-assistants/3.3-tasking.md), where the plan is expanded and the tasks are created via ADO MCP, grounded in the actual repo.

## Activities

### Turn a story into a task checklist

**AI:** Claude / Copilot Chat (grounded Project)
**Context:** User story with acceptance criteria

```
Break this user story into implementation tasks for a Dynamics 365 CE team.
For each task: a one-line title, a 2–3 sentence description a developer can
act on without asking questions, the discipline (config / plugin / web
resource / flow / security / test), and an estimate placeholder.
End with a Definition of Done checklist for the story covering:
build passes, unit tests, solution checker clean, deployed to Dev,
AC demonstrated, documentation updated.
[paste user story]
```

### Sanity-check task completeness

```
Review this task list against the user story and its acceptance criteria.
What is missing? Check specifically for the tasks teams forget:
security role updates, solution/ALM steps, test data setup,
regression impact on existing forms/flows, and documentation.
[paste story + task list]
```

### Normalize task descriptions across the team

```
Rewrite these task descriptions to a consistent format: imperative title,
context sentence, expected outcome, and explicit DoD reference.
Do not change the technical content.
[paste tasks]
```

### Estimation rationale

```
For each user story below, draft an estimation rationale: the main cost
drivers (new tables? plugin logic? integration? migration volume?),
a suggested T-shirt size, and the assumptions that would change the estimate.
Do not present these as final estimates — they are input for the team's
estimation session.
[paste stories]
```

### Push the tasks into ADO via a generated script

```
Convert this feature → task breakdown into a script that creates
the work items in Azure DevOps with the correct parent links.
Target: [az boards CLI / Azure DevOps REST API / CSV for bulk import].
Project: [project name], Area Path: [area], Iteration: [iteration].
Emit the script only; I will review it before running.
[paste breakdown]
```

## Review checkpoints

| Output | What the human checks |
|---|---|
| Task breakdown | Granularity matches team norms (tasks ≤ 1–2 days); no invented scope |
| DoD | Matches the engagement's actual quality gates, not a generic list |
| Estimates | Rationale is input for the team's estimation session, never a final number |
| ADO import script | Dry-run reviewed before execution; work-item types match the process template |
