# [5.1] Validation / Peer Review — design review, test scenarios, and AC cross-checking

> **Matrix cell:** [[5.1]](../ai-augmented-d365ce-activity-matrix.md#51-validation--peer-review) · Column **[1] General-Purpose AI Assistants** · Builds on [B.1](./b.1-initial-setup.md) + [C.1](./c.1-project-setup.md) · Column-3 counterpart: [5.3](../3-coding-assistants/5.3-validation-peer-review.md)

## What this activity is

- **Peer review** on pull requests, review of design docs (at the description/narrative level), check unit-test coverage, sanity-checking an approach, and cross-checking a delivered implementation *narrative* against the requirements doc.

- **Executing test plans** manually from acceptance criteria, may include integration testing, and automated testing -only if it uses distinct tooling from building-.

- **HITL discipline:** and AI review comes *after* the human has already reviewed, as a final detail-catcher — never as the first pass. First-pass AI feedback anchors the human reviewer and defeats the purpose of peer review.

## Activities

### Generate test cases from acceptance criteria

**AI:** Claude / Copilot Chat

```
Convert these Gherkin acceptance criteria into manual test cases
formatted for Azure DevOps Test Plans.
For each scenario, include: Test Case title, preconditions,
step-by-step actions, and expected result.
[paste acceptance criteria]
```

### Generate a UAT script

```
Write a UAT script for [feature] aimed at business users, not testers.
Plain language, one scenario per business task, using the real form and
view names from this Project's knowledge. Include the test data each
scenario needs and a pass/fail checkbox per step.
[paste feature description + ACs]
```

### Review a design document

```
Review this design document as a second reviewer — a human has already
reviewed it. Look for what detail reviews miss: internal contradictions,
requirements from the spec that the design silently drops, unstated
assumptions, and D365 CE anti-patterns (sync logic that should be async,
attribute bloat, security handled in JavaScript).
Report findings only; do not rewrite the document.
[paste or attach design doc]
```

### Cross-check an implementation narrative against requirements

```
Here is the requirements doc / user story, and here is the developer's
summary of what was implemented. List: (1) each AC and whether the
narrative claims to satisfy it, (2) ACs not addressed at all,
(3) implemented behavior that no AC asked for.
[paste both]
```

### Accessibility and UX review for Power Pages

```
Review this Power Pages web template Liquid code for accessibility issues:
missing alt text, poor heading hierarchy, form labels not associated with inputs,
and interactive elements not keyboard accessible.
```

---

## Review checkpoints

| Output | What the human checks |
|---|---|
| Test cases / UAT scripts | Steps executable against the real app; no invented form/field names |
| AI review findings | Triage each finding — the AI flags, the human decides severity and action |
| Sequencing of review | Confirm the human review happened **first**; AI findings appended, not leading |
